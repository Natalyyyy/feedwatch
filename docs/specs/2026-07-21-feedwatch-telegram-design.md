# feedwatch: переименование instawatch + мониторинг Telegram-каналов

Дата: 2026-07-21. Статус: дизайн утверждён Наташей, реализация не начата.

## Зачем

instawatch — скилл-продукт для раздачи: мониторинг чужих публичных
IG-аккаунтов с отчётами в Telegram. Расширяем его на Telegram-каналы,
чтобы пользователь мог следить за постами обеих платформ одним
инструментом. Продукт переименовывается в **feedwatch** — имя перестаёт
врать про «только инста».

Личный трендвотчинг Наташи (`~/Projects/tgstat-puller/watch.py`) остаётся
как есть и на feedwatch не пересаживается.

## 1. Переименование

- Папка `~/Projects/instawatch` → `~/Projects/feedwatch`.
- GitHub-репо `Natalyyyy/instawatch` → `Natalyyyy/feedwatch` (rename в
  настройках репо; GitHub редиректит старые URL и git-remote, но локальный
  remote всё равно обновить на новый адрес).
- Скилл: `~/.claude/skills/instawatch` → `feedwatch`; в SKILL.md имя,
  описание и триггеры расширяются: «что залетело в телеграме», «отчёт по
  каналам», «добавь канал в мониторинг» и т.п. Старые IG-триггеры остаются.
- База: `data/instawatch.db` → `data/feedwatch.db`. Миграция — rename
  файла при первом запуске, если старый есть, а нового нет (у существующих
  пользователей ничего не теряется).
- Все тексты (README, сообщения, комментарии): «мониторинг чужих
  аккаунтов — Instagram и Telegram».

## 2. Конфиг

Вместо плоских `source` + `accounts` — секции по платформам, обе
опциональны (но хотя бы одна обязана быть):

```json
{
  "instagram": { "source": "apify", "accounts": ["natgeo", "nasa"] },
  "telegram":  { "source": "web",   "channels": ["solokumi"] },
  "pulse_multiplier": 2.0,
  "median_window_days": 30,
  "median_min_posts": 12,
  "median_min_age_days": 2,
  "pulse_max_age_days": 3,
  "fetch_limit_pulse": 12,
  "fetch_limit_weekly": 25,
  "weekly_day": "sun",
  "weekly_time": "18:00",
  "pulse_time": "10:00"
}
```

- Общие настройки (медиана, пороги, расписание) — верхний уровень, одни
  на обе платформы.
- `telegram.source`: `web` или `tgstat`. `instagram.source`: `apify` или
  `graph`, как сейчас.
- Каналы в `telegram.channels` — юзернеймы без `@`.
- Миграция старого конфига: при загрузке плоский формат
  (`source`+`accounts` в корне) автоматически трактуется как секция
  `instagram`; `config.example.json` — уже в новом формате.

## 3. База данных

- `posts` получает колонку `platform` (`instagram` | `telegram`).
  Миграция: `ALTER TABLE ... ADD COLUMN platform TEXT NOT NULL DEFAULT
  'instagram'` при подключении, если колонки нет.
- `snapshots` не меняется, колонки переиспользуются:
  - `likes`: IG — лайки, ТГ — сумма реакций;
  - `views`: IG — просмотры видео/рилсов, ТГ — просмотры поста;
  - `comments` — комменты (IG); ТГ-источники счётчик комментов не отдают
    (ни веб-превью, ни лёгкий режим TGStat) → NULL.
- `post_id` для ТГ: `tg:<канал>:<номер поста>` — коллизии с IG-id
  исключены.
- `account` для ТГ — юзернейм канала; `account_status` работает как есть.

## 4. Источники Telegram (fetch.py)

По образцу пары apify/graph добавляются два фетчера.

### `web` — парсинг t.me/s/<канал>

- Без ключей и регистраций, стоимость 0 ₽. Источник по умолчанию для ТГ.
- Страница `https://t.me/s/<канал>` отдаёт ~20 последних постов; для
  `fetch_limit_weekly=25` — вторая страница через `?before=<id>`.
- Из HTML достаём: номер поста, текст, дату (ISO в атрибуте `datetime`),
  просмотры, реакции (сумма эмодзи-счётчиков `tgme_reaction`, платные
  звёзды `tgme_reaction_paid` не считаем), число подписчиков канала,
  пермалинк `https://t.me/<канал>/<id>`. Комментов в превью нет → NULL.
  Проверено на живой странице 2026-07-21.
- Просмотры вида «12.3K» нормализуются в числа.
- Если превью у канала отключено (t.me отдаёт заглушку без постов) —
  канал помечается ошибкой в `account_status`, в отчёте честная строка
  «веб-превью недоступно». То же при HTTP-ошибках.
- Парсинг — стандартной библиотекой + `requests` (как остальной код);
  новых зависимостей не добавляем, HTML разбираем регулярками по
  стабильным классам `tgme_widget_message*` с фикстурой в тестах.

### `tgstat` — TGStat API

- Для тех, у кого есть токен TGStat (`TGSTAT_TOKEN` в `.env`).
- Лёгкий режим: только `channels/posts` (1–2 запроса на канал за прогон,
  `posts/stat` не трогаем — бережём месячный лимит пользователя).
- Клиент — маленький самодостаточный модуль внутри feedwatch (упрощённая
  копия подхода из tgstat-puller/tgstat.py: ретраи, классификация ошибок
  limit/rate/not_found). Зависимости от чужого репо нет.
- В лёгком режиме `channels/posts` отдаёт просмотры, текст, дату и ссылку;
  реакций и комментов там нет → likes/comments NULL. Главная метрика ТГ —
  просмотры — едина для обоих источников.

## 5. Медианы и пульс

- `account_medians` считает медиану и по `views` (третьей к
  likes/comments). Для IG это бонус — рилсы тоже получают медиану
  просмотров.
- Пульс:
  - IG — как сейчас: лайки или комменты ≥ `pulse_multiplier` × медианы;
  - ТГ — триггер по **просмотрам** ≥ `pulse_multiplier` × медианы views
    (реакции показываются в строке сигнала, но не триггерят).
- Правила отстоя медианы (`median_min_age_days`, окно, минимум постов) —
  общие, без платформенных исключений.

## 6. Отчёты

- Weekly и пульс — единые сообщения; внутри секции «Instagram» и
  «Telegram». Настроена одна платформа — одна секция, без пустых
  заголовков.
- Формат строк не меняется: «×2.4 от медианы» + ссылка на пост.
  Для ТГ в топах ранжирование по просмотрам и по реакциям (топ по
  реакциям — только когда данные о реакциях есть, т.е. web-источник).
- В шапке ТГ-секции weekly — текущее число подписчиков каждого канала
  (web: парсится с той же страницы t.me/s; tgstat: из `channels/get`,
  один дополнительный запрос на канал раз в неделю — допустимо).
- Claude-выжимка «о чём писали» — общая на обе платформы;
  `prompts/weekly.md` дополняется: указывать платформу аккаунта.
- Если Claude-вызов не сработал — отчёт без выжимки, как сейчас.

## 7. Setup-флоу скилла

При «настрой мониторинг» SKILL.md ведёт так:

1. Какие платформы: Instagram / Telegram / обе.
2. Для каждой выбранной — источник: IG: apify или graph (как сейчас);
   ТГ: web (по умолчанию, без ключей) или tgstat (нужен токен).
3. Ключи спрашиваются только для выбранных источников; для `web` —
   никаких ключей.
4. Телеграм-бот для отчётов, первые данные, расписание — без изменений.

README: секция стоимости дополняется («Telegram через веб — бесплатно»),
дисклеймер — про ТГ мягче, чем про Мету: веб-превью — публичные данные,
которые Telegram сам отдаёт без авторизации; TGStat — официальный API
по их тарифам.

## 8. Тесты

В стиле существующих (`tests/`, сейчас 48):

- парсер t.me/s/ на сохранённом HTML-фикстуре (обычный пост, пост с
  реакциями/комментами, «12.3K», канал без превью);
- нормализация ответа tgstat;
- миграция конфига (плоский → секции) и базы (добавление `platform`,
  rename db-файла);
- медиана views и платформенная логика пульса;
- формирование отчёта с одной и двумя платформами.

## Вне скоупа

- Сторис и их аналоги.
- История и динамика подписчиков (текущее число в weekly — есть,
  см. раздел 6; накопления и «+N за неделю» — нет).
- Репосты ТГ в отчётах (есть только у tgstat-источника — не выводим ради
  единообразия).
- Пересадка личного trendwatch Наташи на feedwatch.

## Решения, принятые при обсуждении

- Продукт для раздачи → порог входа важнее богатства метрик; но источники
  оба: web (дефолт) и tgstat (по выбору).
- Отчёт единый с опциональными платформенными секциями, не два трека.
- Имя feedwatch (варианты postwatch и pulsewatch отклонены).
